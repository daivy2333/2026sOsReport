https://rust-lang.github.io/rfcs/2394-async_await.html

该 RFC 提议为 Rust 语言引入 `async`/`await` 语法，以改善异步代码编写的人机工程学（ergonomics）体验。核心内容如下：

## 动机

- 高性能网络服务通常需要异步 I/O 支持。Rust 虽已有 `futures` 与 `tokio` 等库，但编写异步代码仍较为繁琐（例如共享状态需要 `Arc` 或链式组合子）。
- `async`/`await` 作为被广泛采用的语法糖，能够使异步代码呈现出与同步代码相近的结构，从而降低认知负担。

## 主要特性

- **异步函数**（`async fn`）：调用时立即返回一个实现了 `Future` trait 的匿名类型，函数体仅在 `poll` 时执行。其返回类型为 `T`（内部类型），而非 `impl Future<Output = T>`（外部类型）。所有输入生命周期均被捕获至返回的 future 中。
- **异步闭包**（`async || {}`）：行为与异步函数类似，调用时返回一个 future。支持 `move` 语义以转移所有权。
- **异步块**（`async {}`）：直接创建 future 的表达式形式。不支持 `return`、`break`、`continue`（除非嵌套于闭包或循环内）。可用于"初始化模式"——在构造时执行同步初始化代码。
- **`await!` 内置宏**：暂停当前 future 的执行，直至被等待的 future 完成；仅在异步上下文中可用。其展开方式类似于循环调用 `poll`，当遇到 `Pending` 时让出控制权。

## 设计细节

- **关键字**：`async` 与 `await` 成为 Rust 2018 edition 的正式关键字。
- **返回类型**：匿名 future 类型实现 `Future<Output = T>`，并具有 `!Unpin` 的负实现（防止移动，因其可能包含跨越 yield 点的引用）。
- **与 `impl Trait` 的区别**：异步函数会捕获所有输入生命周期（相当于返回 `impl Future + 'a`），而非默认的不捕获行为。
- **`async move` 顺序固定**：以避免语法歧义。

## 替代方案与拒绝理由

- **返回外部类型**（如 C# 的 `Task<T>`）会导致生命周期标注极为繁琐，不利于学习和使用。
- **基于生成器（generator）实现**：生成器虽更为通用，但其稳定化路径更长，而 `async` 作为专用语法语义更直观。
- **"热异步函数"（立即执行至第一个 await）**：该方案违背了 Rust future 的惰性求值原则（仅在 `poll` 时执行），容易引发语义混淆。
- **绿色线程等替代模型**：Rust 需要在库层面提供异步机制，而 `async`/`await` 与所有权及借用系统的兼容性最优。

## 未解决问题

- `await` 的最终语法（与 `?` 运算符的优先级交互，以及是否需要分隔符）。
- `for await` 循环处理 `Stream`（留待后续 RFC 解决）。
- 生成器与 `Stream` 的异步生成器语法。
- 允许 `async fn` 实现 `Unpin` 的注解机制（需静态证明不存在跨 yield 点的引用）。
- `async` 块中 `?` 运算符及 `return`、`break`、`continue` 的行为语义（例如是否允许使用 `break` 提前退出块）。

### 后续发展

自 2018 年 RFC 提出至 2026 年的当下，Rust 的异步特性经历了从蓝图规划到逐步成熟的演进过程。以 Rust 1.39 为起点，该语言逐步突破了静态调度层面存在的局限，并通过 2024 Edition 及 2025 年旗舰项目目标的推进，将异步编程体验提升至全新水平。

以下为具体的演进脉络。

### 异步特性演进：从语法稳定到生态完善

#### 起点

- **Rust 1.39（2019 年 11 月）**：具有里程碑意义的版本发布，标志着 `async`/`.await` 语法正式稳定。自此，开发者可在稳定版 Rust 中定义 `async fn` 与 `async` 块，并使用 `.await` 进行等待，Rust 异步生态由此奠定基础。

#### 难点攻坚

- **Rust 1.75（2023 年 12 月）**：稳定了 trait 中的异步函数（AFIT, Async Functions in Traits）。此前受限于编译器实现，`async fn` 无法直接在 trait 中定义，通常需要借助 `#[async_trait]` 宏进行模拟。该功能的稳定使异步 Rust 的静态调度部分摆脱了对宏的依赖，更直接地与 trait 系统集成。
- **Rust 1.67（2023 年 1 月）**：增强了 `#[must_use]` 对 `async fn` 的行为。若异步函数的返回值未被使用或未被 `.await`，编译器将发出警告，帮助开发者避免因遗漏处理 future 而导致的潜在错误。

#### 生产力提升

- **2024 Edition（集成于 Rust 1.85）**：2024 版本对异步编程体验的提升显著，主要包含两大特性：
  - **异步闭包（Async Closures）**：`async ||` 闭包得到稳定支持。这解决了在高阶函数与组合子场景中，使用普通闭包返回 future 时所遇到的类型推导与生命周期难题。
  - **生命周期捕获规则改进**：在 2024 Edition 中，`-> impl Trait`（包括返回 future）默认捕获所有输入生命周期。这显著减少了编写异步代码时原先所需的 `+ '_` 等繁琐生命周期标注，使代码更符合直觉。

#### 未来

- **2025 年及以后的旗舰目标**：Rust 项目在 2025 年的核心目标之一是进一步缩小异步编程与同步编程之间的体验差距。主要包括：
  - **trait 中异步函数的深度优化**：目标是完全替代 `async-trait` crate 的全部功能。为此，团队引入了返回类型标记（RTN, Return Type Notation），解决了长期存在的痛点——如何在 `where` 子句中为 trait 内返回的 `impl Future` 添加 `Send` 等约束。
  - **同步与异步生成器（Generators）**：通过 `iter!` 及类似 `stream!` 的语法，极大简化迭代器与异步数据流的创建，使 `for` 循环风格的异步流处理成为可能。
  - **改善 `Pin` 的人机工程学**：Rust 团队将持续优化 `Pin` 的易用性，降低底层异步编码的学习门槛。

---

### 悬而未决的难题

尽管已取得重大进展，2018 年 RFC 中提出的一些高难度问题至今仍是 Rust 异步演进的核心课题。

- **`async` 块中的控制流与 `?` 运算符**：RFC 曾设想使用 `break` 作为从 `async` 块提前返回的关键字。然而目前，在 `async` 块中直接使用 `return` 将返回整个外层函数，`?` 运算符的行为亦遵循其固有规则。这使得 `async` 块作为独立的可失败计算（fallible computation）使用起来不够直观。`try` 块（try blocks）的推进被认为是解决这一问题的关键一步。
- **异步析构（Async Drop）**：如何处理资源的异步清理（如关闭网络连接）是另一个长期悬而未决的难题。现有的 `Drop` trait 是同步的，无法执行 `.await`。尽管已有 `AsyncDrop` 等提案，其在调度语义、生命周期约束及与现有 `Drop` 的兼容性方面极为复杂，因此尚未被采纳。

### 今日的异步 Rust

总体而言，Rust 的异步支持已从 2018 年的一份蓝图发展为现代系统级异步编程的重要基石。当前的异步体验远不止于语法层面的 `.await`，而是由编译器保证的零成本抽象（zero-cost abstraction）、丰富的异步运行时生态（如 `tokio`、`async-std`）以及持续完善的语言特性共同构成的完整体系。

### 库组件的正式稳定化

本 RFC 提议将第一类 async/await 语法所需的库组件正式稳定化，具体包括：`std::task` 模块的全部 API，以及 `core::future::Future` 与 `std::future::Future`。该提案不涉及 async/await 语法本身的稳定化（将单独进行），亦不包含已另案处理的 `Pin` 相关 API。此 RFC 是对早期 futures RFC 的修订与精简版本，在积累了大量夜间版使用经验后提出。

**动机与背景**：async/await 语法是 Rust 社区最受期待的特性之一，其底层依赖 `Future` 特征及任务系统。稳定这些 API 可使库在稳定版 Rust 上运行，同时无缝支持夜间版的 async/await，并终结 API 层面的设计争议。`Future` 特征的历史可追溯至 2016 年的 futures crate 0.1 版本，此后历经 0.2 版本、Pin API 的引入、以及在 Fuchsia 项目中的大规模实战检验，最终演化为当前标准库中的形态。

**核心 API 设计**：

- `Poll<T>` 枚举：表达异步计算的完成状态，包含 `Ready(T)` 与 `Pending` 两个变体。
- `Waker` 与 `RawWaker`：`Waker` 是任务唤醒句柄，提供 `wake` 方法以通知执行器重新轮询相应任务。为实现执行器自定义唤醒行为，`Waker` 内部封装 `RawWaker`，后者包含一个 `data` 指针和一个 `RawWakerVTable` 虚表（内含 `clone`、`wake`、`drop_fn` 三个函数指针）。该设计避免了 trait 对象的使用，允许灵活的内存管理策略（如引用计数、全局函数等）。`Waker` 满足 `Clone`、`Send`、`Sync`，且要求跨线程调用 `wake` 安全、即使任务已完成亦可安全唤醒、所有实例释放后资源正确回收。
- `Future` 特征：定义方法 `poll(self: Pin<&mut Self>, cx: &Waker) -> Poll<Self::Output>`。使用 `Pin<&mut Self>` 以支持跨 yield 点的借用（自引用结构），`Unpin` 标记 trait 可恢复手动实现时的易用性。`poll` 应快速返回且永不阻塞，未完成时返回 `Pending` 并存储传入的 `Waker`（仅需保存最近一次调用提供的 `Waker`）。任务完成后再调用 `poll` 可能导致 panic 或未定义行为。

**与 futures 0.1 的主要区别**：

1. 使用 `Pin<&mut Self>` 替代 `&mut Self`，以支持异步块内的借用。
2. 移除内置的错误类型（`Error` 关联类型），改为由 `Future` 直接返回 `Result`；必要时可通过 `TryFuture` 特化获得更好的组合子体验。此举使 `async fn` 的语义与普通 `fn` 正交，不捆绑特定的错误处理风格。
3. 显式传递 `Waker` 参数，而非使用线程局部存储（TLS）。这降低了手动实现 future 时的隐式依赖，且对 `no_std` 环境更为友好。

**兼容性**：提供多个兼容层（如 `.compat()` 组合子），使 futures 0.1 与 0.3 可共存，支持生产代码的渐进迁移。

**设计理由与替代方案**：

- **错误类型移除**：改进类型推断，简化 `async fn` 设计（不必强制返回 `Result`），组合子语义更清晰，且与 Rust 的错误处理正交原则一致。其代价是错误密集的代码需额外引入 `TryFuture`，并可能产生混淆。
- **Pin 相关设计**：RFC 选择单一 `Future` 特征（操作 `Pin<&mut Self>`），而非提供 `MoveFuture` 与 `Future` 双特征方案。前者虽对手动实现带来少许不便（需导入 `Pin`/`Unpin`），但具有前向兼容性——未来可在不破坏现有代码的前提下添加 `MoveFuture` 并实现泛化 impl。双特征方案将导致与 `Try` 变种的组合爆炸，且当前编译器支持存在局限。
- **唤醒机制**：早期曾考虑分离 `LocalWaker`（`!Send`）与 `Waker` 以优化单线程执行器性能，但最终合并为单一的 `Waker` 类型。单线程执行器可通过线程局部存储和运行时断言实现类似优化，而统一类型极大地简化了用户 API，并符合多数执行器（如多线程兼容的 tokio）的实践需求。

**未解决问题**：当前 RFC 无悬而未决的问题。

### 语言与生态的持续演进

自 2019 年 `async/.await` 语法在 Rust 1.39 中稳定以来，Rust 的异步特性经历了从基础语法稳定到核心语言能力增强、生态共识形成以及新方向探索的持续演进过程。这一演进在 2023 年至 2025 年间尤为活跃，其核心推动力来自 Rust 项目定期发布的旗舰级目标（Flagship Goals）。

#### 核心语言与库的演进：从稳定到成熟

在核心语言能力和标准库层面，异步 Rust 在此期间实现了多个关键突破：

- **特征中的异步函数（AFIT）稳定**：Rust 1.75 稳定了对特征中 `async fn` 和返回位置 `impl Trait`（RPITIT）的支持。这解决了长期依赖 `#[async_trait]` 宏的问题，使开发者能直接编写 trait 中的异步方法。然而，在公共 trait 中，对于返回 future 的 `Send` 约束等高级功能仍需谨慎处理。

- **2024 Edition 与异步闭包**：作为 Rust 语言发展的重要里程碑，2024 Edition 稳定了异步闭包（async closures）。该特性解决了此前闭包在异步上下文中返回 future 所面临的诸多限制。此外，2024 Edition 还修改了生命周期捕获规则，使 `impl Trait` 返回类型默认捕获所有输入生命周期参数，从而简化了异步函数的签名编写。

- **`Pin` 的人机工程学改进**：针对手动实现 future 或编写底层异步代码时需处理 `Pin` 的难点，Rust 团队在 2025 年持续改进其人机工程学，通过实验性的 `&pin mut T` 语法糖简化了固定引用的创建，降低了编写自引用 `Future` 结构体的门槛。

#### 生态挑战与回应：碎片化与性能

伴随语言的成熟，异步生态在持续演进中也面对并回应着若干核心挑战：

- **运行时与兼容性**：Tokio 已成为事实上的主流异步运行时，但也导致了生态碎片化问题——主流库（如 `reqwest`、`sqlx`）几乎默认与 Tokio 绑定，运行时切换成本高昂。同时，生态系统中存在 futures 0.1 与 0.3 等版本间的兼容层，以支持增量迁移。

- **性能与膨胀问题**：编译器优化不足导致 `async` 函数生成的状态机膨胀，已成为影响生产环境性能的重要因素。已有工程师提交 Rust Project Goal，寻求从编译器层面解决该问题。

#### 持续演进的核心方向

在稳定现有基础后，Rust 项目在 2025 年及之后的工作聚焦于三个旗舰目标：

1. **深化特征中的异步函数**：完全取代 `async-trait` crate 的功能。具体而言，包括实现返回类型标记（RTN, Return Type Notation）以精确约束返回的 future 类型，以及推进动态分发（Dyn Compatibility），通过 `dynosaur` 等 crate 探索 trait 对象的动态分发支持。

2. **推进生成器（Generators）支持**：通过引入 `iter!` 宏等机制，简化同步和异步生成器的创建，旨在为迭代器及异步数据流 `Stream` 提供原生的语言支持。虽然 `iter!` 宏的实验支持已在 nightly 中可用，但其稳定化仍需独立的 RFC 流程。

3. **解决异步析构（Async Drop）问题**：实现异步析构已被列为重要目标之一。尽管已有初步实现草案，其设计仍然复杂，被明确定位为一项原型实验，旨在先行探明实现障碍。距离稳定化仍有大量语言设计工作，目前尚未形成正式的 RFC 草案。
