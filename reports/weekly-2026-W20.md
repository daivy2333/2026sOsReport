## 2. 对原始阻塞串口的分析（Console）

**Console** 是 StarryOS 现有的同步串口驱动，驻留在外部 crate `axplat-riscv64-qemu-virt` 中，不可直接修改。

### 发现的关键特征

- **TX 路径：同步阻塞轮询（polling）** — 发送每个字节时 CPU 空转等待 THR_EMPTY 标志，115200 bps 下每字节约 87 µs 阻塞。
- **RX 路径：中断驱动** — 只使能了 `IER::DATA_READY`（RX 中断），通过 tty-reader 后台协程搬运数据。
- **TX 中断被禁用** — `IER::THR_EMPTY` 未使能，这是 Console 与异步方案的核心矛盾。
- **MMIO 映射在 boot 阶段完成** — 最终页表切换后权限被限制，这是后来发现的深层架构约束。

### Console 带来的约束

- 外部 crate，源码不可修改
- UART 配置状态不透明（无法查询 IIR/MCR/LSR）
- 同步阻塞 TX 与异步 TX 无法共存（数据竞争风险）
- 只使能 RX 中断，不使能 TX 中断

## 3. 对 OS 架构的分析（StarryOS / ArceOS）

深入理解了 StarryOS 的底层架构，特别是与串口相关的部分。

### 关键架构层次

```
axruntime (启动调度)
  → axplat (平台抽象层，外部 crate) — UART MMIO 映射、中断路由
    → axhal (硬件抽象层，外部 crate) — console read/write、register_irq_hook
      → axtask (任务调度，外部 crate) — block_on、poll_io、register_irq_waker
        → kernel (StarryOS 内核) — VFS、设备注册、系统调用
```

### 核心机制理解

- **IRQ waker 机制**：`register_irq_waker(irq, waker)` 使用 `BTreeMap<usize, PollSet>` 管理，支持同一 IRQ 注册多个 waker，但 `register_irq_hook` 全局唯一。
- **VFS 设备注册**：`DeviceOps trait → Device wrapper → File → FD_TABLE` 链路，所有 `/dev` 设备都通过此模式注册。
- **外部 crate 约束**：axplat、axhal、axtask、axpoll 均来自 crates.io，但 kernel 层可通过 `axmm::iomap()` 和直接 `phys_to_virt()` 访问 UART MMIO。
- **启动页表 vs 最终页表**：boot 阶段建立 1G RWX 映射。`axmm::init_memory_management()` 重建页表，但 UART 在 `mmio-ranges` 中，仍被正确映射（READ|WRITE|DEVICE）。**页表不是阻塞原因。**

## 4. 对本地 UART 的分析（uart_16550 v0.6.0）

我们选择了本地路径依赖的 `uart_16550` crate（v0.6.0），而非 crates.io 上的 v0.4.0。

### 关键差异

| 能力 | v0.4.0 (crates.io) | v0.6.0 (本地) |
|------|-------------------|--------------|
| 中断使能控制 | ❌ 缺失 | ✅ `set_interrupt_enable()` |
| 中断类型识别 | ❌ 缺失 | ✅ `interrupt_identification()` + `InterruptType` 枚举 |
| FIFO 触发阈值 | ❌ 缺失 | ✅ `FifoTriggerLevel` 配置 |
| MMIO 后端 | ✅ 有 | ✅ 完善 |

### 核心 API 能力

- `SerialPort::new_mmio(ptr, stride)` — 创建 MMIO 后端实例
- `uart.init(config)` — 完整初始化（波特率、FIFO、中断配置）
- `uart.try_receive()` / `uart.try_send()` — 非阻塞读写
- `uart.isr().interrupt_type()` — 中断类型枚举（ReceivedDataReady / THR_EMPTY 等）
- `uart.set_interrupt_enable(IER)` — 精确控制 RX/TX 中断

这套 API 完全覆盖了我们实现异步串口所需的中断控制能力。问题不在 UART 驱动本身，而在上层架构的访问权限。

## 5. 异步探索：两条路径的尝试

### 方向 A：渐进式集成（feat/uart-async）

**策略**：复用 Console 的 UART 初始化，逐步替换软件路径。

**成果**：
- ✅ M0：基础设施就绪（依赖添加、中断确认）
- ✅ M1：架构验证（Ring Buffer + 中断 + copier 任务模型）
- ✅ M2：VFS 验证（DeviceOps + 设备注册 + poll/epoll）

**失败点 — M3 替换**：
- 尝试将 Console 底层替换为 AsyncUart 时，出现 **IRQ 风暴 + TX busy-loop**
- Console 的 UART 状态与 AsyncUart 不兼容（IER 配置冲突）
- 未在替换前验证 UART 硬件状态（IIR/MCR/LSR）
- THR_EMPTY 状态理解错误（库注释与实际行为不一致）
- 最终回滚，转向新策略

### 方向 B：完全剔除 Console（feat/uart-async-dev2）

**策略**：从零开始，使用本地 uart_16550 独立初始化 UART，完全不依赖 Console。

**成果**：
- ✅ P0：项目规划与设计完成
- ⚠️ P1：UART 硬件初始化替代 — 发现 **MMIO 权限问题**（内核上下文无法访问 UART 寄存器）

**阻塞点 — P2**：
- 尝试在 ISR 中访问 UART 寄存器 → **同样 LoadFault**
- 证明 MMIO 权限限制对**所有上下文**（内核 + ISR）都生效
- 验证了关键结论：**不彻底更改底层支持就无法使用异步串口**

### MMIO 权限的误区（05-31 纠正）

此前认为的方向 B P1/P2 "MMIO 权限阻塞"经过 05-31 代码验证后证实为误判：

- axconfig.toml 的 mmio-ranges 明确包含 UART（0x10000000, 4K）
- new_kernel_aspace() 将其正确映射为 READ|WRITE|DEVICE
- Console 的 MmioSerialPort 能访问恰好证明了映射有效
- **真正的根因**: `UART_STRIDE=4` — NS16550 寄存器仅 0x00-0x07 共 8 字节，stride=4 把 ISR（offset 2×4=8）读到了 base+8，超出寄存器范围 → 总线错误 → LoadFault
- 修复: stride 改为 1 后，内核 + ISR 全部正常

## 6. 遇到的问题汇总

| 问题 | 发现阶段 | 状态 | 说明 |
|------|---------|------|------|
| **IRQ 风暴** | 方向 A M3 | ❌ 失败 | Console 与 AsyncUart IER 冲突 + stride=4 |
| **TX busy-loop** | 方向 A M3 | ❌ 失败 | TX FIFO 满，UART 状态异常 |
| **Console 状态不兼容** | 方向 A M3 | ⚠️ 待解决 | Console 只使能 RX 中断，AsyncUart 需 TX 中断 |
| **stride=4 LoadFault** | 方向 B P1/P2 | ✅ 已修复 | NS16550 仅 8 字节，stride=4 越界。改 1 即可 |
| ~~MMIO 权限（内核）~~ | 方向 B P1 | ⚠️ 误判 | 页表映射正常，非真正阻塞 |
| ~~MMIO 权限（ISR）~~ | 方向 B P2 | ⚠️ 误判 | 同上，同一 stride 错误 |

### 关键教训

1. **硬件集成前验证配置常量** — stride=4 浪费了两个方向的大量时间
2. **Console 能工作是 proof** — MmioSerialPort stride=1 始终正常，本身就是页表正常的证据
3. **现象相似≠根因相同** — LoadFault 可能是页表、stride、PMP、地址错误等多种原因
4. **THR_EMPTY 的库注释有误导性**

### Q1 实现的架构

```

           ├─ RX: disable_rx_intr() → RX_WAKER.wake()
           └─ TX: disable_tx_intr() → TX_WAKER.wake()

RX copier (bg task)              TX copier (bg task)
  poll_fn: UART.read FIFO          poll_fn: buf.pop_tx
           buf.push_rx                      UART.write THR

           RX_WAKER.register                Poll::Pending
           Poll::Pending
```

### 关键突破：stride=4 根因发现

之前方向 B P1/P2 的全部 LoadFault 阻塞，根因是一个常量：

```
UART_STRIDE: u8 = 4  →  ❌ ISR at base+8 越界 → LoadFault
UART_STRIDE: u8 = 1  →  ✅ 全部正常
```

NS16550 是字节寻址设备，寄存器空间仅 0x00-0x07。stride=4 下 offset 2（ISR）× 4 = 第 8 字节，超出 UART 范围。QEMU 返回总线错误，RISC-V 解释为 LoadFault。

### 已废弃的方案

此前基于"MMIO 权限阻塞"错误前提提出的 A/B/C/D 四方案均不再适用：

- ~~方案 A: Polling TX~~ — stride=1 修复后不需要
- ~~方案 B: 修改 axplat~~ — 页表映射正常，不需要改
- ~~方案 C: 回退 Console~~ — AsyncUart 可独立实现
- ~~方案 D: 内核页表修复~~ — 页表本身没问题
