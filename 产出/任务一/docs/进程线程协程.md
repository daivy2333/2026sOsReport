# 进程、线程与协程概念解析

## 基于爬虫程序的具体实现分析

以下通过我们实现的三种爬虫程序（代码见 `产出/任务一/爬虫任务/crawler_rs/`），分析进程、线程、协程三种调度方式在实际代码中的表现特征。

### 进程爬虫

```rust
// process_crawler.rs (核心逻辑)
let child = Command::new(exe)
    .arg("worker")
    .arg("--name").arg(school.name)
    .arg("--cache-dir").arg(&cache_dir)
    .stdout(Stdio::piped())
    .spawn()?;
children.push(child);

// ... 等待所有子进程完成
for mut child in children {
    child.wait()?;
    // 从 stdout 解析结果
}
```

**实现特征**：
- 每个 URL 启动一个独立的 OS 子进程（`std::process::Command`），子进程运行与父进程相同的二进制文件，通过 `worker` 子命令区分角色。
- 父子进程通过 **stdout 管道**（`Stdio::piped()`）进行进程间通信（IPC）。子进程将结果以 `|` 分隔的文本行输出，父进程逐行解析。
- 父进程调用 `child.wait()` **阻塞等待**每个子进程退出，确保所有任务完成。
- 每个子进程拥有 **独立的地址空间**，包括页表、文件描述符表、Rust 运行时堆。子进程启动时读取 `/proc/self/status` 报告 VmRSS，父进程汇总所有子进程的内存占用。

**理论映射**：这是最纯粹的进程级并行。`fork(2)`/`exec(2)` 的开销体现在：复制父进程地址空间、加载动态链接器、初始化 Rust 运行时（分配器、标准库线程本地存储等）。

### 线程爬虫

```rust
// thread_crawler.rs (核心逻辑)
let (tx, rx) = mpsc::channel();

for school in SCHOOLS {
    let tx = tx.clone();
    let cache = cache_dir.to_path_buf();
    handles.push(thread::spawn(move || {
        let record = process_task(school.name, &cache, cpu_repeat, io_repeat);
        tx.send(record).ok();
    }));
}

for handle in handles {
    handle.join().ok();
}
```

**实现特征**：
- 每个 URL 启动一个 **OS 线程**（`std::thread::spawn`），所有线程共享同一进程的地址空间。
- 线程间通过 **`mpsc::channel`**（多生产者单消费者通道）收集结果。每个线程独立拥有 `tx` 发送端的克隆，向同一个接收端发送数据。`mpsc` 内部使用互斥锁保证线程安全。
- 主线程通过 `handle.join()` **阻塞等待**每个线程结束，然后通过 `rx.try_recv()` 收集所有结果。
- 每个线程拥有 **独立的函数调用栈**（默认 2MB 虚拟空间，实际提交仅数 KB），但堆内存（包括文件数据、Future 对象等）在线程间天然共享。

**理论映射**：线程共享地址空间，创建开销远小于进程（仅需分配栈和线程控制块，无需复制页表）。`mpsc::channel` 的 `send` 操作涉及锁竞争，在大量线程同时发送时可能成为瓶颈。`thread::spawn` 是内核级操作，每次调用都涉及 `clone(2)` 系统调用。

### 协程爬虫

```rust
// async_crawler.rs (核心逻辑)
let semaphore = Arc::new(Semaphore::new(concurrency));

for school in SCHOOLS {
    let sem = semaphore.clone();
    let cache = cache_dir.to_path_buf();
    handles.push(tokio::spawn(async move {
        let _permit = sem.acquire().await.unwrap();
        process_task(school.name, &cache, cpu_repeat, io_repeat)
    }));
}

for handle in handles {
    if let Ok(record) = handle.await {
        records.push(record);
    }
}
```

**实现特征**：
- 每个 URL 提交一个 **tokio 异步任务**（`tokio::spawn`），所有任务在 tokio 的线程池上协作式调度。默认线程数等于 CPU 核心数。
- 通过 **`Semaphore`**（信号量）限制最大并发数，`sem.acquire().await` 在资源可用前**挂起当前任务但不阻塞 OS 线程**——这是协程区别于线程的关键行为。被挂起的任务让出线程去执行其他就绪任务。
- 主任务通过 `handle.await` **异步等待**每个任务完成。与线程的 `join()` 不同，`await` 不会阻塞线程——如果被等待的任务尚未完成，当前协程让出执行权。
- 所有任务在 tokio 的**工作窃取（work-stealing）调度器**上运行。当一个工作线程空闲时，它会从其他线程的任务队列中"偷取"任务执行，实现负载均衡。
- 任务状态机由 Rust 编译器自动生成：`async` 块被编译成实现了 `Future` trait 的匿名状态机，跨 `.await` 的局部变量存储在状态机结构体中（堆上）。

**理论映射**：这是协作式调度的典型实现。`Semaphore` 的 `acquire().await` 是一个自然的 yield 点——当资源不可用时，任务主动让出 CPU。tokio 的工作窃取调度器实现了用户态的负载均衡，无需内核参与。所有任务共享 OS 线程的栈空间，仅在执行时暂用，让权时释放。

### 三种实现的核心差异总结

| 维度 | 进程爬虫 | 线程爬虫 | 协程爬虫 |
|------|---------|---------|---------|
| 创建单元 | `Command::new().spawn()` | `thread::spawn` | `tokio::spawn` |
| 通信方式 | stdout 管道（IPC） | mpsc 通道（共享内存+锁） | await + 共享状态 |
| 等待机制 | `child.wait()` 阻塞 | `handle.join()` 阻塞 | `handle.await` 非阻塞 |
| 并发控制 | 全部同时启动（无上限） | 全部同时启动（无上限） | Semaphore 显式限流 |
| 地址空间 | 每个子进程独立 | 所有线程共享 | 所有任务共享 |
| 栈 | 每个进程独立 | 每个线程独立 | 共享线程栈 |
| 切换方式 | 内核抢占 | 内核抢占 | 用户态协作（await） |
| 系统调用 | fork/exec/wait/pipe | clone/join/futex | 无（纯用户态调度） |

## I/O 接口类型

I/O 接口按同步/异步和阻塞/非阻塞两个维度分类：

| 类型 | 示例 | 特点 |
|------|------|------|
| 同步阻塞 | read | 阻塞当前线程直到完成，每连接需独立线程 |
| 同步非阻塞 | read(NONBLOCK) | 未就绪立即返回，需配合多路复用 |
| 多路复用 | select/poll/epoll | 单线程处理全部 I/O，阻塞等待事件发生 |
| 异步 | MPI_iread | 分提交/完成两步，可在等待期间执行计算 |
| 循环队列 | io_uring | 共享内存中请求/完成双队列，生产消费并行 |

性能依次提升，易用性依次下降。本质是请求状态与执行状态的解耦程度：二者越松耦合，并行性越高，但易用性越差。

## 事件机制：中断与轮询

**中断**：事件发生时打断当前执行流，产生临时执行流运行中断处理函数，完成后返回原执行流。适用于事件不频繁但实时性要求高的场景。中断处理过程中一般不允许嵌套，处理过程应尽量短。

**轮询**：当前执行流永不被中断，需要主动询问目标事件是否发生。适用于事件频繁、要求高并发的场景——此时若开启中断会造成大量切换开销。

中断的根源是 CPU 在指令粒度上对中断信号的轮询。CPU 可将中断分发到不同特权级处理，但主流实现是：先分发到 OS，OS 再通过信号等机制转发到用户程序。

## 任务管理基础

**任务**：由若干相互调用的函数组成、具备一定功能的指令执行流，是 CPU 调度的最基本单位。

### 任务执行上下文环境

- **地址空间**（进程）：决定任务能访问的内存范围
- **特权级**（进程/线程）：决定任务能执行的指令
- **任务状态**：运行态（占用 CPU）、就绪态（等待 CPU）、等待态（等待事件）
- **函数调用栈**
  - **线程**：在整个生命周期占有固定栈
  - **协程**：仅执行时占用栈，主动让权时释放

### 任务切换方式

- **抢占式**：随时可能切换（时钟中断），保存状态多，切换开销大，保证实时性
- **协作式**：调用特定操作时切换（yield），保存状态少，切换开销小，不保证实时性

## 进程、线程与协程详解

### 进程

每个进程有独立的地址空间，因此有页表切换开销。内核可以是一个独立的进程（有自己独立的地址空间），也可以是用户进程地址空间中的一部分。在异步操作系统中，内核作为独立进程时，系统调用过程变成一种特殊的进程切换。

### 线程

每个线程有独立的函数调用栈，线程切换时需要保存和恢复全部寄存器。当内核与用户线程不在同一地址空间时，每个用户线程只有用户栈（无对应内核栈），每个内核线程只有内核栈（无对应用户栈）。

### 协程（无栈协程）

可理解为状态机转移函数，执行时可共用同一个函数调用栈。在编译器或函数库的支持下，编译时将 async 函数变换成状态机变迁和同步函数的组合。异步函数中跨越 await 的变量存放在 Future 对象中（一般在堆上），其它变量放在栈上或寄存器中。

协程在主动让权时会出让处于空状态的函数调用栈；但由于中断或异常导致的执行态协程暂停时，无法回收占用的函数调用栈，从而成为一个线程。

### 三种维度对比

| 维度 | 进程 | 线程 | 协程（无栈） |
|------|------|------|------------|
| 地址空间 | 独立 | 共享 | 共享 |
| 栈 | 独立 | 独立 | 共享（仅执行时占用） |
| 切换发生时机 | 抢占 | 抢占 | 协作（await/yield） |
| 切换开销 | 最大（页表+TLB+上下文） | 中（内核上下文切换） | 最小（函数调用级） |
| 创建开销 | ms 级 | μs 级 | ns 级 |
| 保存状态 | 全部寄存器+地址空间 | 全部寄存器 | 仅跨 await 变量 |

### 概念映射关系

传统模型向协程模型的映射：

- 1 个进程 + 多个线程分别运行在不同 CPU 上 → 1 个进程 + 多个协程，每个协程属于不同线程
- 1 个进程 + 多个线程运行在同一 CPU 上 → 1 个进程 + 多个抢占式协程，所有协程同属一个线程

由于实时性需求，应同时保留抢占式和协作式协程/任务，关键问题是两种协程调度的统一——切换时根据中断方式（被动/主动）自动判断保存/恢复哪些状态。

## 协程与线程对比

无栈协程相对于线程（或有栈协程）的优势：

1. **切换开销更小**：协程只可能在有限的 await 语句处主动让权，需要保存的状态信息较少。主动让权时的协程切换无需切换函数调用栈，访存局部性更好。协程切换相当于一次函数返回和调用，寄存器的使用遵守 ABI 调用约定，一般不会涉及全部寄存器的保存和恢复。

2. **任何执行流都可能在任意指令处被中断**（如时钟中断），此时需要保持整个函数调用栈的占用——该执行流实际上是一个线程。

3. **理想的切换统一**：对于调度器而言，协程和线程没有本质区别。事实上，在调度线程看来，执行一个线程也是一次函数调用（两次上下文切换被包装成 switch 函数），更接近于协程。

## 统一的上下文切换层级

极致的状态是：应用程序的主动切换一定是协程切换；在统一了进程、线程和协程的管理数据结构后，可以在协程切换中选择不同进程和不同线程中的下一个就绪协程。

- **协程切换（栈复用）**：同一进程同一线程内，函数调用级切换
- **线程切换（栈切换）**：被抢占时的切换或主动让权后选择了占有函数调用栈的任务，需要执行用户态线程库代码，进行用户栈保存和切换
- **进程切换（特权级切换）**：不同进程间的协程切换，需要内核参与进行地址空间、用户堆栈和协程控制块的切换

## 异步编程模式

### yield 模式

应用程序调用 yield 函数放弃当前协程的 CPU 使用权，由调度器选择下一个协程。实现简单，与语言关系不大。

```
fn yield(rip, state) {
    let current = Task { rip, state };
    let next = QUEUE.pop();
    QUEUE.push(current);
    // 切换到 next
}
```

### poll 模式

事件循环驱动：每次从队列取出一个事件，调用其 poll 方法，返回 Ready/Pending。Ready 则移除，Pending 重新入队。一般与语言相关（Rust, Node.js）。

```
loop {
    let task = QUEUE.pop();
    match task.poll() {
        Ready => {},
        Pending => QUEUE.push(task),
    }
}
```

### 两种模式的统一

- yield 模式可用 yield() 函数实现
- poll 模式可用 yield 关键字实现（将 yield 替换为下一次调用的起始点）

底层使用 yield 函数作为统一的调度模型更为灵活。

## 异步系统调用

### 基于同步系统调用的异步实现

目前的系统调用都是同步的，即使 io_uring 也是用两次同步系统调用来完成异步效果：

1. **第一次系统调用**：创建用户进程与内核进程的共享内存，建立请求/响应队列
2. **第二次及之后的系统调用**：查询前一次结果（不进内核），或在无结果且有新请求时仅加入请求队列

### 理想的异步系统调用

编译时自动生成系统调用请求代码和协程控制块数据结构。第一次调用进内核，中间各次只查询结果和切换协程，最后一次结果返回时使用事件通知机制。

### 同步 vs 异步系统调用执行过程

**同步**：用户进程发起请求→阻塞→内核执行服务→返回→恢复用户进程。全程两次上下文切换。

**异步**：通过共享内存的请求队列和响应队列实现：

1. 用户将请求填入队列尾端，首次后非阻塞通知内核
2. 内核依次从队列首端取出请求、执行
3. 内核将响应填入响应队列尾端，首次后非阻塞通知用户
4. 用户依次从队列首端取出响应、恢复对应协程

中间过程无用户/内核切换，只有对共享内存的轮询。

### 同步与异步的关系

- 同步系统调用：用户进程→阻塞→内核执行→返回
- 异步系统调用：分两步（提交+等待完成），中间可执行其他计算
- 两者的切换代码均由内核提供，用户进程只能执行

## 统一调度展望

协程、线程、进程的调度可以统一在同一框架下：

- 同一线程内协程切换 → 函数调用（编译器生成的状态机）
- 同一进程内线程切换 → 用户态线程库（保存/恢复用户栈）
- 不同进程间切换 → 内核参与（地址空间切换）

这需要：
- 统一的就绪任务队列（协程和线程共用）
- 统一的任务切换机制（依据是否在同一线程/进程内分情况选择切换方式）
- 统一的事件注册和分发机制
- 内核与用户态的调度状态共享（通过 vDSO 等机制）

## RISC-V 用户态中断

用户态中断（N 扩展）允许用户进程直接收发中断，无需内核参与。当前标准仍在起草阶段。

### 应用场景

- 进程间通信（IPC）：目标进程在本核时直接发中断，不需切换到内核
- 信号机制的优化：由用户态信号服务例程处理的信号可完全在用户空间完成
- 外设请求处理：如按键事件直接递送到用户进程的中断处理例程

### 与 Cache 结合的中断处理流程

1. 中断发生时用中断编号查专用 Cache
2. Cache 命中 → 直接跳转到用户态中断处理例程
3. Cache 未命中 → 陷入内核，内核查询后更新 Cache，下次命中

### 多核场景

中断转发由核间的中断转发模块处理，传递的信息包括：核 id、进程 id、中断信息。转发过程无需内核参与，仅需硬件支持。

## 内核异步化相关研究工作

### 内核的进程化改造

将操作系统内核改为一个独立进程，将系统调用视为进程间通信的特例。内核地址空间和用户进程地址空间被视为两个独立地址空间。


### 异步运行时打通用户态与内核态

设计目标：

1. 统一的协程调度：内核提供调度器代码，使用一致的维护数据结构，但调度器代码的执行分布在用户态和内核态
2. 进程内的协程切换在用户态执行（开销更小）
3. 进程切换在内核态执行特权指令
4. 用户态和内核态通过共享内存（RingBuffer）通信，尽量减少用户/内核切换
5. 低功耗：若用户未用到内核功能，内核不占用任何 CPU 资源

### 性能分析

异步内核的性能提升主要体现在：

- 内核栈数目的减少
- 内核态协程切换开销小于传统内核线程切换
- 对多线程 I/O 密集型应用效果较明显

若要实现更大的性能提升，需要设计新的系统调用接口来打通用户态与内核态的运行时，这意味着全新的用户程序开发方式。
