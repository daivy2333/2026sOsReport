# 绿色线程实现分析

## 概念

绿色线程（green thread）是一种由用户态运行时管理的线程，操作系统内核对其存在一无所知。与内核线程（1:1 模型）不同，绿色线程采用 M:N 模型——M 个绿色线程复用在 N 个内核线程上执行。

在 Rust 发展历史上，绿色线程曾被作为标准并发原语纳入，但在 1.0 版本发布前被移除。官方给出的理由是其不是零成本抽象（zero-cost abstraction），且与 C 语言的 FFI 兼容性存在问题。

## 实现框架

以下分析基于 `产出/任务二/green-thread/` 项目的实现。

### 核心数据结构

```rust
struct Thread {
    id: usize,
    stack: Vec<u8>,          // 独立栈空间（2MB）
    ctx: ThreadContext,      // CPU 寄存器上下文
    state: State,            // 线程状态
    task: Option<Box<dyn Fn()>>,
}

enum State {
    Available,  // 空闲可用
    Ready,      // 就绪等待调度
    Running,    // 正在执行
}

struct ThreadContext {
    rsp: u64,   // 栈指针
    r15-rbp: u64, // 被调用者保存的寄存器
}
```

### 状态机

每个绿色线程在生命周期中经历 `Available → Ready → Running → Available/Ready` 的状态变迁：

- **Available**：线程槽位空闲，可分配给新任务
- **Ready**：任务已分配，等待 CPU
- **Running**：正在执行任务
- 任务完成后回到 Available；主动 yield 后回到 Ready

### 上下文切换

上下文切换通过保存和恢复 CPU 寄存器实现，是绿色线程最核心的机制：

```rust
// 保存当前线程的寄存器到 old，从 new 恢复寄存器
unsafe fn switch(old: *mut ThreadContext, new: *const ThreadContext) {
    naked_asm!(
        "mov [rdi + 0x00], rsp",   // 保存栈指针
        "mov [rdi + 0x08], r15",   // 保存被调用者保存寄存器
        // ...
        "mov rsp, [rsi + 0x00]",   // 恢复新线程的栈指针
        // ... 恢复其他寄存器
        "ret"
    );
}
```

此实现使用 `#[unsafe(naked)]` 函数（不生成函数序言/结尾），直接在汇编中操作寄存器。保存和恢复的寄存器集合符合 System V AMD64 ABI 中被调用者保存的寄存器约定。

### 栈初始化

为新任务创建执行环境时，需要在分配给它的栈上预先布置调用帧：

```
栈顶高地址
┌──────────────┐
│  guard 地址   │  ← 任务返回时跳转到 guard
├──────────────┤
│  skip 地址    │  ← 填充对齐
├──────────────┤
│  f (函数地址)  │  ← 任务入口点
├──────────────┤  ← RSP 初始指向这里
│              │
│  可用栈空间    │
│              │
└──────────────┘
栈底低地址
```

当任务执行完毕（`f()` 返回）时，`ret` 指令弹出 `skip` 地址，`skip` 立即 `ret`，再弹出 `guard` 地址，进入 `guard` 函数。`guard` 负责将线程状态设为 Available 并通过 `t_yield` 调度下一个线程。

## 调度策略

当前实现采用 **Round-Robin** 调度：从当前线程位置开始，顺序查找下一个 Ready 状态的线程。每个线程获得公平的 CPU 时间。

```rust
fn t_yield(&mut self) -> bool {
    let mut pos = self.current;
    while self.threads[pos].state != State::Ready {
        pos += 1;
        if pos == self.threads.len() { pos = 0; }
        if pos == self.current { return false; }
    }
    // 切换到 pos 线程
}
```

> 本实现还支持**最高优先级优先**调度扩展，详见 [`green-thread/priority-extension.md`](../green-thread/priority-extension.md)。

## 执行流状态变迁

绿色线程在执行过程中的状态变迁：

| 事件 | 旧状态 | 新状态 | 触发条件 |
|------|--------|--------|---------|
| spawn | Available | Ready | 分配任务 |
| 调度选中 | Ready | Running | `t_yield` 选择 |
| yield | Running | Ready | 主动让出 CPU |
| 任务完成 | Running | Available | `guard` 函数执行 |
| 所有线程完成 | - | - | `t_yield` 返回 false |

## 局限性与改进方向

1. **协作式调度**：线程必须主动调用 `yield_thread()` 才能让出 CPU。不支持抢占式调度（无时钟中断）。
2. **固定线程数**：`MAX_THREADS` 硬编码，限制了可并发执行的任务数量。
3. **单线程执行**：所有绿色线程在同一个 OS 线程上运行，无法利用多核。
4. **未处理阻塞 I/O**：如果一个绿色线程执行了阻塞系统调用，整个进程会阻塞，其他绿色线程无法运行。
