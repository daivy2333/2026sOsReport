## 1. 当前串口调度策略

StarryOS 当前只有一个串口接口：`/dev/console`，由 N_TTY 实现，底层通过 `axhal::console` 操作 UART 16550 硬件。

### 1.1 接收路径（已中断驱动，但开销大）

```
UART IRQ (0x0a) → PLIC → S_EXT trap → axhal::irq_handler
  → register_irq_waker 唤醒 tty-reader copier 任务
  → Console.read_bytes (从硬件 FIFO 批量读取)
  → N_TTY ldisc 处理 (行规则、termios、信号处理)
  → 写入 N_TTY 内部 buffer
  → 用户态 read(/dev/console) 从 buffer 读取
```

**关键点**：
- ✅ **接收路径已中断驱动** — 使用 `register_irq_waker` + `tty-reader` 后台任务
- ⚠️ **但经过多层处理** — Console → N_TTY → ldisc → termios → 用户态，每层有开销
- ⚠️ **行规则开销** — 即使 raw 模式，数据仍经过 ldisc 处理路径

### 1.2 发送路径（同步阻塞，CPU 空转）

```
用户态 write("/dev/console") → N_TTY.write
  → Console.write_bytes (axhal::console)
  → 循环等待 THRE 标志 (忙等待！)
  → 写入 THR 寄存器
```

**关键代码** (`axplat-riscv64-qemu-virt/console.rs`):

```rust
pub fn write_bytes(bytes: &[u8]) {
    for &byte in bytes {
        while !uart.line_sts().contains(LineStsFlags::OUTPUT_EMPTY) {}
        uart.send_raw(byte);
    }
}
```

**问题**：
- ❌ **同步阻塞循环** — 每字节都等待 THRE 标志（TX FIFO 空）
- ❌ **CPU 空转** — 发送期间 CPU 无法执行其他任务
- ❌ **波特率越低阻塞越长** — 115200 bps 下发送 1 KB 需要 ~87 ms CPU 空转

### 1.3 当前架构总结

| 路径 | 调度方式 | 优点 | 问题 |
|------|---------|------|------|
| **RX 接收** | 中断驱动 + copier 任务 | 无 CPU 空转，低延迟 | 多层处理开销，行规则开销 |
| **TX 发送** | 同步阻塞忙等待 | 简单可靠 | CPU 空转，阻塞时间长 |
| **VFS 接口** | DeviceOps + FileLike | 标准化 | 无独立缓冲区，依赖 N_TTY |

## 2. 同步阻塞瓶颈分析

### 2.1 发送路径阻塞点

| 阻塞场景 | 阻塞位置 | 阻塞时长估算 | 影响 |
|----------|---------|-------------|------|
| **内核 printf/日志输出** | `axhal::console::write_bytes` | 每条日志数 ms ~ 数百 ms | 调试信息输出阻塞 CPU，影响实时性 |
| **用户态 write("/dev/console")** | N_TTY.write → write_bytes | 取决于数据量和波特率 | shell 命令输出阻塞用户态 |
| **批量数据传输** | write_bytes 循环 | 115200 bps 下 1 KB ≈ 87 ms | 无法高效传输数据 |
| **多个 write 调用连续执行** | 每个 write 都阻塞 | 累积阻塞时间成倍增长 | 系统吞吐量低 |

**实测阻塞时间**（理论估算）：

```
波特率 115200 bps:
  发送 1 字节: ~86.8 µs CPU 空转
  发送 1 KB:   ~87 ms CPU 空转
  发送 64 KB:  ~5.5 s CPU 穽转

波特率 1 Mbps:
  发送 1 KB:   ~10 ms CPU 空转
  发送 64 KB:  ~640 ms CPU 穽转
```

### 2.2 接收路径开销点（非阻塞但有延迟）

| 开销场景 | 开销位置 | 开销类型 | 影响 |
|----------|---------|---------|------|
| **N_TTY 层处理** | `Tty<Console, Console>` | 结构体封装、方法调用 | 每次读/写都有虚调用开销 |
| **ldisc 行规则** | `terminal/ldisc.rs` | termios 处理、信号检测、特殊字符处理 | 即使 raw 模式仍需经过处理分支 |
| **信号处理开销** | ldisc 检测 Ctrl+C 等特殊字符 | 条件判断、信号发送 | 每次 read 都需检查 |
| **多层 buffer 复制** | Console buffer → N_TTY buffer → 用户态 | 数据多次复制 | 内存带宽浪费 |

### 2.3 其他阻塞点

| 场景 | 阻塞位置 | 问题 |
|------|---------|------|
| **poll/select 无数据时** | 用户态等待 rx 事件 | 需等待 tty-reader 填充 buffer，无主动通知 |
| **高波特率下 IRQ 风暴** | 中断频繁触发 | RX FIFO 触发阈值小（默认 1 字节）导致中断过于频繁 |
| **多端口并发** | 无支持 | 只有一个 `/dev/console`，无法多端口并发 |

## 3. 优化方向与原因

### 3.1 发送路径异步化

**目标**: 消除 CPU 空转，实现零阻塞发送。

**原因**:
1. **CPU 空转浪费资源** — 87 ms 发送 1 KB 时 CPU 完全空转，无法执行其他任务
2. **实时性要求** — 内核日志输出不应阻塞系统其他功能
3. **高吞吐量需求** — 数据传输场景需要高效率

**方案**:
- 引入 TX ring buffer + TX copier 任务
- 用户态 write → 写入 tx_buf → 立即返回
- TX copier 从 tx_buf 读到硬件 FIFO，使能 TX 中断
- TX IRQ → 唤醒 TX copier → 继续发送

### 3.2 接收路径简化（raw 模式零开销）

**目标**: 提供独立 `/dev/ttyS0` 设备，raw 模式零行规则开销。

**原因**:
1. **N_TTY 多层开销** — 即使 raw 模式，数据仍经过 ldisc 处理路径
2. **应用场景需求** — 数据传输场景需要纯粹的字节流，不需要 termios
3. **灵活性** — 可切换 termios，但默认 raw 零开销

**方案**:
- AsyncUart 默认 raw 模式，直接读写 ringbuf
- termios 可通过 ioctl TCGETS/TCSETS 动态启用
- 启用 termios 时复用现有 ldisc 逻辑，但处理在 ringbuf 层

### 3.3 中断驱动优化

**目标**: 减少中断频率，提高吞吐量。

**原因**:
1. **中断风暴风险** — RX FIFO 默认触发阈值 1 字节，高波特率下中断过于频繁
2. **PLIC 处理开销** — 每次 IRQ 都需 PLIC claim/complete，MMIO 开销
3. **ISR → Waker → 任务切换** — 每次中断都有上下文切换开销

**方案**:
- 配置 FIFO 触发阈值（14 字节），减少中断频率
- NAPI 风格批量轮询：中断触发后切换到轮询模式处理 FIFO 残留
- 批量 API：`try_receive_batch` / `try_send_batch` 减少 MMIO 次数

### 3.4 多端口并发支持

**目标**: 支持 4 端口并发，每个端口独立缓冲区和 copier 任务。

**原因**:
1. **扩展性需求** — 嵌入式系统常需多串口（调试、数据传输、外设通信）
2. **并发吞吐量** — 多端口并发可提高系统总吞吐量
3. **资源隔离** — 每个端口独立缓冲区，避免相互影响

**方案**:
- AsyncUart trait 抽象硬件操作
- 每个 UART 实例独立 rx_buf + tx_buf + copier 任务
- 设备注册为 `/dev/ttyS0`, `/dev/ttyS1`, `/dev/ttyS2`, `/dev/ttyS3`

### 3.5 空闲 CPU 零占用

**目标**: 无数据传输时 CPU 进入休眠，零空转。

**原因**:
1. **功耗要求** — 嵌入式系统需要低功耗
2. **资源浪费** — CPU 空转浪费计算资源
3. **实时性冲突** — 空转占用 CPU，影响其他实时任务

**方案**:
- copier 任务在无数据时 poll_fn 返回 Pending，挂起任务
- axtask 调度器无任务可执行时 CPU 进入 WFI 休眠
- 中断到来时唤醒任务，CPU 恢复执行

```
         │                    │

    UART 16550 MMIO       AtomicWaker
         │                    │
         ▼                    ▼
    ISR (中断服务例程)    UART IRQ → wake()
```

### 4.2 数据流

**RX 接收路径**:

```
UART IRQ → ISR
  ↓
  1. 读 IIR 判断中断源 (RX 就绪)
  2. 禁用 RX 中断 (避免风暴)
  3. AtomicWaker::wake() 唤醒 RX copier
  ↓
RX copier (poll_fn 循环)
  ↓
  1. 从 UART FIFO批量读取 (try_receive_batch)
  2. 写入 rx_buf (HeapRb)
  3. rx_wakers.wake() 唤醒等待的读者
  4. 若 FIFO 还有数据 → 继续读 (NAPI 批量轮询)
  5. FIFO 空后使能 RX 中断 → 返回 Pending 挂起
  ↓
用户态 read("/dev/ttyS0")
  ↓
  block_on(poll_io(...)) 从 rx_buf 读取
```

**TX 发送路径**:

```
用户态 write("/dev/ttyS0")
  ↓
  写入 tx_buf → tx_wakers.wake() 唤醒 TX copier
  ↓
TX copier (poll_fn 循环)
  ↓
  1. 从 tx_buf批量读取
  2. 写入 UART FIFO (try_send_batch)
  3. FIFO 满后使能 TX 中断 → 返回 Pending 挂起
  ↓
UART IRQ (TX 空) → ISR
  ↓
  1. 读 IIR 判断中断源 (TX 空)
  2. 禁用 TX 中断 (避免风暴)
  3. AtomicWaker::wake() 唤醒 TX copier
  ↓
TX copier 继续发送
```

### 4.3 ISR 极简原则

```rust
// ISR (中断服务例程) 只做三件事
fn uart_isr() {
    let iir = uart.interrupt_identification();
    match iir.interrupt_type() {
        InterruptType::RxReady => {
            uart.disable_rx_intr();  // 禁用 RX 中断
            rx_waker.wake();         // 唤醒 RX copier
        }
        InterruptType::TxEmpty => {
            uart.disable_tx_intr();  // 禁用 TX 中断
            tx_waker.wake();         // 唤醒 TX copier
        }
        _ => {}
    }
}
```

**所有数据搬运都在 copier 任务中完成**，ISR 无竞态，无需锁。

### 4.4 VFS集成

```rust
impl DeviceOps for UartAsyncDriver {
    fn read_at(&self, offset: usize, buf: &mut [u8]) -> AxResult<usize> {
        block_on(poll_io(&self.rx_wakers, IoEvents::IN, || {
            self.rx_buf.pop_slice(buf).ok_or(AxError::WouldBlock)
        }))
    }

    fn write_at(&self, offset: usize, buf: &[u8]) -> AxResult<usize> {
        block_on(poll_io(&self.tx_wakers, IoEvents::OUT, || {
            self.tx_buf.push_slice(buf).ok_or(AxError::WouldBlock)
        }))
    }

    fn as_pollable(&self) -> Option<Arc<dyn Pollable>> {
        Some(self.clone())  // 支持 poll/select/epoll
    }
}
```

## 5. 实现路线图

### 5.1 Milestone 规划

| Milestone | 目标 | 预期收益 |
|-----------|------|---------|
| **M0** | 基础设施就绪（uart_16550 依赖 + Embassy 集成 + QEMU 双串口） | 开发环境可用 |
| **M1** | 中断驱动串口可用（Ring Buffer + ISR + copier 任务） | 发送路径异步化，消除 CPU 空转 |
| **M2** | 异步 API + VFS 集成（DeviceOps + /dev/ttyS0） | 用户态可读写，poll/epoll 可用 |
| **M4** | 性能优化（批量传输 + NAPI + 空闲零占用） | 吞吐量达标，CPU 零空转 |
| **M3** | Console 统一（替换 Console 底层为 AsyncUart） | 内核日志异步化，消除 printf 阻塞 |
| **M5** | 真板验证（VisionFive2） | 真实硬件验证 |
| **M6** | DMA 探索（远期） | 极高吞吐量场景 |

### 5.2 推进策略

```
M0 → M1 → M2（串行推进，必做）
              │
              ├→ M4（优先）← 在稳定独立串口上优化性能
              │      ├→ M5（真板验证）
              │      └→ M6（DMA 远期）
              │
              └→ M3（延后）← Console 统一风险最高，在 M4 稳定后再做
```

**风险控制**:
- M3（Console 统一）涉及修改内核启动核心路径，风险最高
- M4（性能优化）在独立串口上推进，有清晰基准
- 先验证性能再改核心路径，降低集成风险

## 总结

**当前问题**:
- TX 发送路径同步阻塞 → CPU 空转，阻塞时间长
- RX 接收路径虽中断驱动但多层开销 → 延迟和吞吐量受限
- 无独立缓冲区和硬件抽象 → 不灵活，不可扩展

**优化方案**:
- ISR → AtomicWaker → copier 任务架构 → TX 异步化，RX 简化
- Ring Buffer + PollSet → 独立缓冲区，支持多等待者
- AsyncUart trait → 硬件抽象，多端口并发
- NAPI 批量轮询 + FIFO 触发优化 → 减少中断频率，提高吞吐量

**预期收益**:
- CPU 空转消除 → TX 阻塞时 CPU 可执行其他任务
- 吞吐量提升 → 估算 5x ↑，可达 90% 线速
- 延迟降低 → RX 延迟 P50 < 500 µs
- 多端口并发 → 4 端口独立运行
- 空闲零占用 → 无数据时 CPU 进入休眠
