# 优先级调度对比分析：绿色线程 vs Tokio 异步运行时

## 1. 实验背景与设计

任务一实现了进程、线程、协程三种并发模型的爬虫，对33所高校HTML页面进行爬取并通过多级负载参数系统性地对比了三种模型的性能。核心结论是协程在I/O密集型场景下优势显著（中量I/O下比线程快22%，轻量I/O下快66%），而纯CPU密集型负载下三种模型吞吐率基本相当，差异在6%以内。任务二在绿色线程运行时基础上拓展了优先级调度能力，新增`spawn_with_priority(f, priority)`接口和`find_highest_priority_ready()`算法，通过`SchedulerMode`枚举（RoundRobin / HighestPriorityFirst）切换调度策略，并以13个自动化测试验证了优先级调度的正确性。

任务三将两个任务的工作负载和调度器连接起来，让爬虫的CPU密集型校验和计算在优先级调度器上运行。核心验证问题是：**在纯CPU满载场景下，用户态协作式优先级调度能否让高优先级任务获得更多CPU时间，以及这种效果在多大程度上依赖任务的主动合作。**

33所高校按索引分为三个批次，每个批次绑定一个优先级值：

| 批次 | 学校索引 | 优先级值 |
|------|---------|---------|
| 高优批（PRIO=2） | 0-10（北大、清华等11所） | 2 |
| 中优批（PRIO=1） | 11-21（重庆大学、复旦大学等11所） | 1 |
| 低优批（PRIO=0） | 22-32（武汉大学、四川大学等11所） | 0 |

每批由一个独立的绿色线程执行（`spawn_with_priority`创建，分别挂载到Thread 1/2/3）。在`workload_checksum`函数的每100次外层迭代处注入`yield_thread()`调用，使CPU密集型任务在执行约100轮校验和后主动让出CPU，让调度器有机会重新选择最高优先级的就绪线程。

负载设计覆盖6种场景：

| 负载类型 | cpu_repeat | io_repeat | 说明 |
|---------|-----------|-----------|------|
| 无负载 | 0 | 0 | 基线：纯缓存读取，无额外负载 |
| 轻量CPU | 200 | 0 | 约2次yield/校，适合快速验证 |
| 满载CPU | 10000 | 0 | 约100次yield/校，核心测试场景 |
| 轻量I/O | 0 | 100 | 每校101次系统调用 |
| 中量I/O | 0 | 1000 | 每校1001次系统调用 |
| 饥饿测试 | 1000 | 0 | yield关闭，演示协作式局限 |

对照实验采用A/B设计：相同负载下运行**实验组（HighestPriorityFirst）**和**对照组（RoundRobin）**，排除HTML页面大小差异等负载本身特征的影响。Tokio对比实验增加两组配置——**Tokio Priority**（BinaryHeap分发器实现优先级）和**Tokio Default**（无优先级基线，`--base-priority 0`），构成4组调度配置的完整对比矩阵。

## 2. 绿色线程调度实验

### 2.1 Priority vs RoundRobin：核心对比

实验在32核Linux机器上执行，cpu_repeat=10000，io_repeat=0。两种调度模式的前三笔调度trace对比如下：

**Priority模式**：
```
[ 0.025] Thread 0 → Ready, selecting Thread 1 (Ready → Running, prio=2)
[ 6.317] Thread 1 → Ready (yield), selecting Thread 2 (Ready → Running, prio=1)
[11.316] Thread 2 → Ready (yield), selecting Thread 1 (Ready → Running, prio=2)
```

**RoundRobin模式**：
```
[ 0.035] Thread 0 → Ready, selecting Thread 1 (Ready → Running, prio=2)
[ 5.851] Thread 1 → Ready (yield), selecting Thread 2 (Ready → Running, prio=1)
[10.817] Thread 2 → Ready (yield), selecting Thread 3 (Ready → Running, prio=0)
```

差异在第3笔调度中显现：Priority模式下Thread 2 yield后调度器选中Thread 1（prio=2），Thread 3（prio=0）永远不会被选中，只要Thread 1或Thread 2中至少有一个处于Ready状态。RoundRobin模式下按顺序选中Thread 3，三个线程公平轮转。

Priority模式下完整的调度模式是两个较高优先级线程交替执行：Thread 1（prio=2，运行约5-6ms）→ yield → Thread 2（prio=1，运行约4-5ms）→ yield → Thread 1。Thread 3（prio=0）在此过程中完全被饿死——它只有在Thread 1和Thread 2都执行完毕后才会被调度。

两种模式的总执行时间非常接近，验证了CPU负载总量相同：

| 指标 | Priority模式 | RoundRobin模式 |
|------|------------|---------------|
| 总执行时间（用户态） | 14.99s | 14.81s |
| 总平均延迟 | 895.7ms | 1447.9ms |
| 高优批平均延迟 | 1111.9ms | 1345.7ms |
| 低优批平均延迟 | 308.3ms | 1510.3ms |
| 完成模式 | 高/中优先交替→低优最后 | 三批均匀交错 |

Priority模式下低优批的延迟（308ms）显著低于高优批（1112ms）——低优批在缓存预热后无竞争地运行，这是优先级调度的次级效应。RoundRobin模式下三个批次延迟相近（约1345-1510ms），无优先级偏差。

### 2.2 特殊场景验证

**I/O负载场景**：当cpu_repeat=0、io_repeat=100/1000时，`workload_checksum_yielding`循环次数为0，不会触发任何yield。但这不意味着优先级调度失效——批次函数在处理每个学校时仍通过`process_task_yielding`执行。由于绿色线程在执行`fs::read`时不会yield，每个学校的所有I/O重复在读完成后才继续下一学校。因此批次内11所学校连续处理，批次间遵循优先级顺序：高优批全部完成→中优批全部完成→低优批全部完成。RoundRobin模式下三批任务均匀交错。

**同优先级基线**：设置`--base-priority 0`使所有批次以相同优先级创建。在HighestPriorityFirst模式下，`find_highest_priority_ready()`选择第一个匹配的Ready线程（最低thread index）。Thread 1和Thread 2交替执行、Thread 3被饿死，行为与Priority模式一致。这确认了优先级效果由`spawn_with_priority`传入的优先级值决定，而非批次函数名称或学校顺序引起。同优先级下线程索引成为隐含的次级排序依据。

**饥饿演示（关闭yield注入）**：通过`--no-yield`标志完全禁用`yield_thread()`调用。Thread 1一次性完成全部11所学校的CPU校验和计算，从未yield，期间Thread 2和Thread 3完全没有获得任何CPU时间。三个线程完全串行化。这展示了协作式调度的核心局限：优先级调度的有效性完全依赖任务的主动合作。

## 3. Tokio 异步调度对比

### 3.1 实现与性能

Tokio不提供原生的任务优先级支持，其默认调度器采用work-stealing策略，无优先级概念。本实验在Tokio上实现了优先级调度分发器：通过`BinaryHeap`按优先级排序待处理任务，分发器循环弹出最高优先级任务并调用`tokio::spawn`提交执行；通过`tokio::sync::Semaphore`控制并发度。

四种调度配置在CPU满载（cpu_repeat=10000）下的对比：

| 调度器 | 配置 | 优先级机制 |
|--------|------|-----------|
| green-thread-priority | `--scheduler green-thread --mode priority` | `find_highest_priority_ready()` |
| green-thread-rr | `--scheduler green-thread --mode roundrobin` | 轮转 |
| tokio-priority | `--scheduler tokio --base-priority 2` | 优先级分发器 |
| tokio-default | `--scheduler tokio --base-priority 0` | 无优先级（FIFO） |

| 调度器 | 用户态时间(s) | 实际耗时(s) | 总均延迟(ms) |
|--------|-------------|------------|-------------|
| GT Priority | 14.99 | 15.01 | 895.7 |
| GT RoundRobin | 14.81 | 14.80 | 1447.9 |
| Tokio Priority | 20.43 | **6.35** | 704.3 |
| Tokio Default | 21.17 | **4.52** | 729.4 |

Tokio实际耗时仅为绿色线程的30-42%，多核并行是主要优势（用户态CPU时间×4核≈实际耗时）。Tokio Default略快于Tokio Priority（4.52s vs 6.35s），BinaryHeap分发器增加了约2s的调度开销。

在Tokio Priority模式下，高优先级任务全部完成后低优先级任务才开始执行，完成顺序与绿色线程优先级模式一致。但执行模式存在关键差异：绿色线程中高优和中优批次交替执行，Tokio分发器一次性提交高优批次的所有任务到worker池，高优批次内部任务可并发执行（受concurrency限制），而非交替。

### 3.2 综合对比

完整调度器对比总结：

| 维度 | green-thread-priority | green-thread-rr | tokio-priority | tokio-default |
|------|----------------------|-----------------|----------------|---------------|
| 调度模型 | 协作式用户态 | 协作式用户态 | 协作式 async | 协作式 async |
| 优先级机制 | `find_highest_priority_ready()` | 无 | BinaryHeap 分发器 | 无 |
| 并发能力 | 单线程 | 单线程 | 多 worker 并行 | 多 worker 并行 |

## 4. 分析与结论

### 4.1 优先级生效的机制分析

关键机制是yield注入。在`workload_checksum_yielding`函数的每100次外层迭代处调用`yield_thread()`，使每个CPU密集型任务在执行约100轮校验和后主动让出CPU，调度器`t_yield()`被调用并扫描所有Ready状态的线程。

关键行为细节在`t_yield()`的实现中：`find_highest_priority_ready()`在当前线程变为Ready之前执行，因此yield的线程不会与自己竞争。当Thread 1（prio=2）yield时，寻找最高优先级Ready线程时看到Thread 2（prio=1）和Thread 3（prio=0），选中Thread 2。当Thread 2 yield时，Thread 1已经处于Ready（prio=2），因此被重新选中。Thread 3（prio=0）永远不会被选中，直到Thread 1和Thread 2都完成。

如果没有yield注入，每个线程在执行10000轮校验和时完全占用CPU，Thread 1会一次性完成所有11所学校才让出CPU，期间所有其他线程被饿死。这正是协作式调度的理论边界：在缺乏硬件中断的环境中，调度器无法强制剥夺正在运行的任务。突破这一边界需要进入内核态，通过定时器中断实现时间片轮转，或通过硬件中断优先级实现真正的抢占式调度。

### 4.2 调度器选择的影响

绿色线程和Tokio代表了用户态调度的两种范式。绿色线程是完全自定义的调度器，拥有对线程状态和上下文切换的完全控制，适合教学和调度算法实验，但不具备生产级的并发性能和多核利用能力。Tokio是工业级异步运行时，提供高效的多核work-stealing调度，缺乏原生的优先级支持，但可以通过分发器模式在应用层实现优先级调度。

### 4.3 结论

1. **协作式优先级调度有效但依赖yield注入**。通过在计算密集循环中注入yield点，高优先级任务在CPU满载场景下获得优先执行权。每100次校验和迭代yield一次产生了约4-6ms的有效时间片。

2. **无yield时优先级调度完全失效**。关闭yield注入后，第一个获得CPU的线程会一直运行到完成，所有其他线程被饿死。这反映了协作式调度的理论边界。

3. **同优先级对照实验确认了优先级效果的因果关系**。当所有批次设为相同优先级时，行为模式与Priority模式一致而非RoundRobin模式，证明差异来自调度策略而非数据特征。

4. **Tokio通过分发器模式实现优先级调度，执行效率是绿色线程的3倍**。Tokio Priority在CPU满载下实际耗时6.35s（绿色线程15.01s），多核并行是主要优势。分发器引入约2s调度开销。

5. **跨调度器对比验证了优先级调度效果的普遍性**。无论底层调度器是自定义绿色线程还是Tokio work-stealing，通过适当的优先级机制都可以实现任务优先级的差异化处理。
